This Software Design Description (SDD) was prepared by Group M of CS342 - F1 as a partial fulfillment of the 
course requirements for Software Engineering 1 at Cebu Institute of Technology ‚Äì University. It serves as the 
primary design reference for AmyLens, an on-device rice grain image validation and amylose classification 
system designed to operate entirely within a local area network. 
This document translates the previously approved AmyLens Software Requirements Specification (SRS) into a 
detailed component-level design blueprint covering all four modules of the system: the Android mobile application 
(Modules 1 and 2), the Spring Boot backend server (Module 3), and the React web dashboard (Module 4). Each 
module is designed and documented by its assigned team member. 
The design decisions reflected in this document represent the team's agreed-upon architecture as of the time of 
submission and are considered final. This document is intended for review by the course instructor and for internal 
reference by all members of Group M throughout the development phase.

1.‚Äã Introduction‚Äã
 
 
Purpose 
 
The purpose of this document is to translate the AmyLens Software Requirements Specification (SRS) into a 
complete and detailed design blueprint for the entire system ‚Äì an on‚Äëdevice rice grain image validation, grain 
extraction, amylose classification, and local server‚Äëbased anomaly detection system that operates entirely 
within a local area network (LAN). It defines the architectural structure, component interactions, and detailed 
transaction‚Äëlevel design for all four modules (Android mobile app, Spring Boot backend, and React web 
dashboard). This document serves as the definitive reference for developers, the team lead, and stakeholders 
to ensure alignment with all functional requirements, non‚Äëfunctional performance targets, and interface 
contracts. The externally observable behaviour is fully specified: a researcher captures a rice grain image on 
an Android tablet, selects their name and session metadata, and within 5 minutes receives an amylose class 
(Waxy/Low/Intermediate/High) along with a confidence score, while the team lead reviews flagged sessions 
on a web dashboard that never sends data outside the LAN. 
 
 
Scope 
 
This document covers the complete design of the AmyLens system, which includes: 
 
‚óè‚Äã
An Android mobile application (Modules 1 and 2) ‚Äì designed for on‚Äëdevice image validation, grain 
segmentation, amylose classification, metadata selection, offline queuing, and session submission to the 
local server. 
‚óè‚Äã
A local Spring Boot backend server (Module 3) ‚Äì designed to handle device registration, session ingestion, 
automated anomaly detection (using mirrored GQ‚ÄëRIS historical data), dashboard data serving, session 
review actions, data export (CSV, JSON, PDF), and reference data import. 
‚óè‚Äã
A React web dashboard (Module 4) ‚Äì designed for team lead authentication, device management, flagged 
session review, analytics visualisation, and standardised data export. 
‚óè‚Äã
All communication ‚Äì designed to occur exclusively over the institution‚Äôs WiFi LAN using HTTPS (TLS), with 
no internet egress. The physical deployment is assumed to be contained within a single branch station or 
research facility. 
 
The SDD explicitly exclude: 
 
‚óè‚Äã
Any cloud‚Äëbased processing, external API calls, or communication beyond the LAN. 
‚óè‚Äã
Integration with laboratory wet‚Äëchemistry equipment or external grain analysers. 
‚óè‚Äã
Real‚Äëtime video processing or batch processing of multiple images simultaneously. 
‚óè‚Äã
User authentication beyond device‚Äëbased approval (SSAID) and a single hardcoded admin dashboard 
login (bcrypt‚Äëhashed). 
‚óè‚Äã
Deployment procedures, hardware provisioning, or user training. 
‚óè‚Äã
Any changes to the underlying classifier model ‚Äì the SDD treats the amylose classifier as a fixed black‚Äëbox 
component.

‚ñÝ‚Äã
Restart the browser.

Firefox 
‚ñÝ‚Äã
